## 新增需求

### 需求:所有页面必须使用翻译函数
所有 UI 文本必须通过翻译函数获取，禁止硬编码文本。

#### 场景:上传页面使用翻译
- **当** 用户访问上传页面
- **那么** 所有 placeholder、按钮文本来自翻译文件

#### 场景:活动列表页面使用翻译
- **当** 用户访问活动列表页面
- **那么** 活动类型、按钮文本、提示消息来自翻译文件

#### 场景:海报创建页面使用翻译
- **当** 用户访问海报创建页面
- **那么** 模板描述、按钮文本、提示消息来自翻译文件

#### 场景:海报详情页面使用翻译
- **当** 用户访问海报详情页面
- **那么** alert、confirm 消息来自翻译文件

#### 场景:管理后台页面使用翻译
- **当** 管理员访问后台页面
- **那么** 角色标签、状态标签、导航文本来自翻译文件

#### 场景:表盘相关页面使用翻译
- **当** 用户访问表盘相关页面
- **那么** 分类标签、状态标签、表单文本来自翻译文件

### 需求:翻译文件必须包含所有必要的 key
zh-CN.json 和 en.json 必须包含所有页面使用的翻译 key。

#### 场景:翻译文件同步
- **当** 添加新的翻译 key
- **那么** 两个语言文件都必须包含该 key

#### 场景:翻译 key 命名规范
- **当** 添加翻译 key
- **那么** key 使用层级结构命名（如 pageName.element.type）

## 修改需求

### 需求:系统必须支持中文和英文两种语言
系统必须提供中英文双语支持，用户可以切换语言。（保持不变，扩展覆盖范围）

#### 场景:默认语言
- **当** 用户首次访问系统
- **那么** 系统使用中文作为默认语言

#### 场景:切换到英文
- **当** 用户点击语言切换并选择英文
- **那么** 页面所有 UI 文本切换为英文

#### 场景:切换到中文
- **当** 用户点击语言切换并选择中文
- **那么** 页面所有 UI 文本切换为中文

#### 场景:所有页面响应语言变化
- **当** 用户切换语言
- **那么** 所有页面的 UI 文本都正确更新
