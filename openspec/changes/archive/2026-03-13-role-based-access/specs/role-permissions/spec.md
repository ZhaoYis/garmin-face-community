## 新增需求

### 需求:系统必须支持四种用户角色
系统必须支持 guest（游客）、user（普通用户）、creator（创作者）、admin（管理员）四种角色。

#### 场景:默认角色
- **当** 新用户首次登录
- **那么** 系统自动分配 user 角色

#### 场景:角色存储
- **当** 用户被分配角色
- **那么** 角色信息存储在 user.role 字段中

### 需求:系统必须为每个角色定义权限范围
每个角色必须有其明确的功能权限边界。

#### 场景:游客权限
- **当** 未登录用户访问系统
- **那么** 用户可以浏览海报模板、查看公开海报，但无法绑定 Garmin、生成海报

#### 场景:普通用户权限
- **当** 角色为 user 的用户登录
- **那么** 用户可以绑定 Garmin、生成海报、分享海报

#### 场景:创作者权限
- **当** 角色为 creator 的用户登录
- **那么** 用户拥有普通用户所有权限，并可上传表盘作品、管理自己的作品

#### 场景:管理员权限
- **当** 角色为 admin 的用户登录
- **那么** 用户拥有所有权限，并可访问后台管理、模板管理、用户管理

### 需求:系统必须提供权限检查机制
API 和页面必须能够检查当前用户是否有权限访问。

#### 场景:页面权限检查
- **当** 用户访问需要特定权限的页面
- **那么** 系统检查用户角色，无权限时重定向到 403 页面

#### 场景:API 权限检查
- **当** 用户调用需要特定权限的 API
- **那么** 系统检查用户角色，无权限时返回 403 错误
