## 上下文

当前系统使用 NextAuth.js 进行认证，用户表已有 `role` 字段，但仅支持 `admin` 和 `user` 两个值。需要扩展为四层角色体系，并在整个应用中实现基于角色的访问控制。

### 角色定义

| 角色 | 权限 |
|------|------|
| guest | 浏览海报模板、查看公开海报 |
| user | 绑定 Garmin、生成海报、分享海报 |
| creator | 上传表盘作品、管理作品 |
| admin | 后台管理、模板管理、用户管理 |

## 目标 / 非目标

**目标：**
- 实现四层角色体系和权限检查
- 创建管理员后台（用户管理、模板管理、表盘审核）
- 实现创作者表盘上传和管理功能
- 为所有受保护的 API 路由添加权限检查

**非目标：**
- 不实现细粒度的资源级权限（如"只能编辑自己的海报"）
- 不实现角色继承或自定义角色
- 不实现权限变更的审计日志

## 决策

### 1. 角色存储

使用数据库 `user.role` 字段存储，类型为 `enum('guest', 'user', 'creator', 'admin')`。

**替代方案：**
- 使用数字等级（1-4）- 不够直观
- 使用关联表存储权限 - 过于复杂

### 2. 权限检查策略

采用**中间件 + 辅助函数**的双重检查：
- `middleware.ts` - 页面级别的路由保护
- `requireRole()` 辅助函数 - API 级别的权限检查

**替代方案：**
- 仅使用中间件 - API 路由无法保护
- 使用装饰器模式 - Next.js 不支持

### 3. 管理员后台路由

使用 `/admin/*` 路由前缀，由中间件统一保护。

### 4. 创作者功能路由

使用 `/watchfaces/*` 路由，创作者可在该页面上传和管理表盘。

## 风险 / 权衡

- **游客身份识别**：游客无账号，无法在数据库存储 → 使用 session 判断，未登录用户视为 guest
- **权限提升攻击**：用户可能尝试访问更高权限的页面 → 中间件严格检查 + API 二次验证
- **性能影响**：每次请求都需检查权限 → 权限信息存储在 session 中，无需额外查询

## 迁移计划

1. 更新数据库 schema，扩展 role 枚举
2. 运行迁移，将现有 `user` 角色保持不变
3. 部署新的中间件和权限检查逻辑
4. 创建管理员后台页面
