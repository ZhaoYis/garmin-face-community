## 新增需求

### 需求:用户可以同步 Garmin 运动数据
系统必须允许已绑定 Garmin 的用户从 Garmin API 同步运动记录。

#### 场景:手动同步
- **当** 用户点击"同步数据"按钮
- **那么** 系统调用 Garmin API 获取最近的运动记录，存储到本地数据库，并显示同步结果

#### 场景:增量同步
- **当** 用户已有历史数据再次同步
- **那么** 系统仅获取上次同步时间之后的新记录，避免重复存储

### 需求:系统必须存储运动记录的完整信息
运动记录必须包含活动类型、时间、距离、配速、心率、海拔等数据。

#### 场景:跑步记录存储
- **当** 同步到跑步类型的运动记录
- **那么** 系统存储 activityType=running, durationSeconds, distanceMeters, avgPaceSeconds, avgHr, maxHr, elevationGain 等字段

#### 场景:骑行记录存储
- **当** 同步到骑行类型的运动记录
- **那么** 系统存储 activityType=cycling 及相关数据

#### 场景:游泳记录存储
- **当** 同步到游泳类型的运动记录
- **那么** 系统存储 activityType=swimming 及相关数据

### 需求:系统必须存储运动路线 Polyline
对于包含 GPS 数据的运动记录，系统必须存储 polyline 编码的路线数据。

#### 场景:路线数据存储
- **当** 运动记录包含 geoPolylineDTO
- **那么** 系统存储 polyline 字段用于后续海报路线渲染

### 需求:系统必须处理同步失败
当 Garmin API 调用失败时，系统必须提供清晰的错误信息。

#### 场景:API 限流
- **当** Garmin API 返回 429 错误
- **那么** 系统显示"请求过于频繁，请稍后重试"的提示

#### 场景:Token 失效
- **当** refresh_token 也失效无法刷新
- **那么** 系统提示用户重新绑定 Garmin 账号

### 需求:同步运动数据必须要求用户已登录
只有已登录的用户才能同步 Garmin 运动数据。

#### 场景:未登录用户同步
- **当** 未登录用户尝试调用同步 API
- **那么** 系统返回 401 未授权错误

#### 场景:已登录用户同步
- **当** 已登录用户点击"同步数据"按钮
- **那么** 系统开始从 Garmin API 同步运动记录
